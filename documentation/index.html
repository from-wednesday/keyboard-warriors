Keyboard Warriors — JSON & System Reference

Schema definition, token vocabulary, chip rendering rules, and system component glossary.

JSON Schema

Each shortcut entry in an app JSON file uses the following fields. The shortcut field from the old schema is removed. mac_sc and pc_sc are now token strings used for both matching logic and chip rendering.

// Full entry — all fields shown { "mac_sc": "CMD+SHIFT+Z", // token string for Mac matching + chip rendering "pc_sc": "CMD+SHIFT+Z", // token string for PC matching + chip rendering "display": "Redo", // plain text shown as the question "difficulty": 2, // integer 1–5 "group": "history", // optional — only when pool de-duplication needed "keyboard_conflict": true // optional — omit when false; presence = conflict exists } // Minimal entry — Mac only shortcut, no group, no conflict { "mac_sc": "CONTROL+SHIFT+SLASH", "display": "Open Keyboard Shortcuts", "difficulty": 3 } // Platform-divergent shortcut { "mac_sc": "CMD+OPTION+M", "pc_sc": "CMD+CONTROL+M", "display": "Add to Render Queue", "difficulty": 3 }

Field rules

mac_sc / pc_sc — Token string. Omit the field (or set to null) if the shortcut does not exist on that platform. At least one must be present.

display — Plain text only. No markup.

difficulty — Integer 1 to 5. Controls which pool the shortcut enters based on the player's difficulty setting.

group — Optional short string. The pool builder caps any named group at two questions per ten-question set. Only add when genuinely needed (e.g. opacity shortcuts 1–9).

keyboard_conflict — Optional boolean. Omit entirely when false. When present and true, the shortcut is excluded from the pool for physical keyboard users and only appears for touch-enabled users.

Token string rules

Tokens are separated by +.

Order does not matter for matching or rendering — the system canonicalises order automatically before display.

Case does not matter — the system uppercases all tokens internally. Uppercase is recommended for readability.

Canonical display order for chips is always:
CMD → CONTROL → OPTION → SHIFT → key on Mac
WIN → CMD → OPTION → SHIFT → key on PC

Chip Rendering

Chips appear in the report card, cheat sheet, toast hints, end screen history, and conflicts settings. Two sizes are used depending on context.

Canonical chip order

Regardless of the order tokens appear in the JSON string, chips always render in this order:

⌘→ ⌃→ ⌥→ ⇧→ key Mac: CMD → CONTROL → OPTION → SHIFT → key
⊞→ Ctrl→ Alt→ ⇧→ key PC: WIN → CMD → OPTION → SHIFT → key

System Component Glossary

Use these names when discussing the system. Each corresponds to a distinct function in the JavaScript.

NormaliseToken( token )
Token normaliser
Converts any accepted synonym to its canonical form. ALT → OPTION, ENTER → RETURN, CTRL → CMD etc. Called first by everything else so the rest of the system only ever sees canonical tokens.
ParseShortcutString( shortcutString )
String parser
Splits a token string on +, runs each token through NormaliseToken, and returns a structured object: which modifier flags are required (cmd, shift, option, control, win) and what the key token is. Example: CMD+SHIFT+Z → { cmd: true, shift: true, option: false, control: false, win: false, key: 'Z' }.
GetActiveShortcutString( shortcutEntry )
OS selector
Takes a shortcut JSON entry object and returns the correct token string for the current OS. Returns entry.mac_sc on Mac, entry.pc_sc on PC. Returns null if the field is absent or null (shortcut does not exist on this platform).
Replaces all direct reads of entry.shortcut throughout the codebase.
ShortcutMatchesEvent( shortcutString, keyboardEvent )
Match checker
Takes a token string and a browser keyboard event. Returns true if the event satisfies the shortcut. Calls ParseShortcutString internally, then checks modifier flags and physical key code against the parsed result.
Replaces matchesShortcut().
ShortcutToKeyCodes( shortcutString )
Keyboard lighting
Takes a token string and returns the array of browser key code strings that need to be lit on the graphical keyboard display. Used only by the lighting system, not by match checking.
Replaces shortcutToCodes().
RenderShortcutChips( shortcutString, isLongForm )
Chip renderer
Takes a token string and a boolean. When isLongForm is true (single-token shortcut), renders wide chips with word labels. When false (multi-token), renders compact chips with symbols only. Canonicalises token order before rendering. Returns a DOM element.
Replaces BuildShortcutKeyChips().

Migration Notes

While migrating the JSON files

Keep the existing shortcut field in place alongside the new mac_sc and pc_sc token strings until the new HTML is deployed. The old code still reads shortcut. Once the parser rewrite is live, the old field can be removed from all entries.

Three malformed entries to fix now

Three entries in the current files have tokens that were concatenated without separators. These need fixing regardless of the migration:

ALTCTRLN → OPTION+CMD+N
ALTCTRLO → OPTION+CMD+O
ALTCTRLP → OPTION+CMD+P

Check which file these appear in and verify the intended shortcuts before correcting.
